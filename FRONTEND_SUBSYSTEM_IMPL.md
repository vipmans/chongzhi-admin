## 1. 基于 `api.json` 的前端架构结论

当前 `api.json` 只支持两套前端登录边界：

### 1.1 管理平台

对应接口：

- `POST /admin/auth/login`
- `GET /admin/auth/me`

结论：

- 内部员工统一登录一个管理平台
- 登录后根据角色和权限展示不同菜单和功能
- 不能再拆出多个内部独立登录平台

### 1.2 渠道管理平台

对应接口：

- `POST /portal/auth/login`
- `GET /portal/me`

结论：

- 外部渠道统一登录一个渠道管理平台
- 登录后只能访问本渠道数据

## 2. 前端必须遵守的架构要求

必须保持：

- 一个管理平台前端
- 一个渠道管理平台前端

明确禁止：

- 再拆成多个内部员工独立登录站点
- 用多个内部平台代替角色菜单控制

正确做法：

- 管理平台内部按角色显示不同菜单和功能
- 渠道管理平台保持单独前端

---

## 3. 管理平台前端实现要求

## 3.1 角色和权限不能混用

- 角色：岗位身份
- 权限：功能点

前端用户态建议至少保留：

- `userId`
- `username`
- `displayName`
- `roleCodes` 或 `primaryRoleCode`
- `permissionCodes`
- `menuCodes`
- `dataScope`
- `accessToken`
- `refreshToken`

## 3.2 菜单与页面控制

管理平台登录成功后，根据角色与权限动态控制：

- 左侧菜单
- 页面访问权限
- 按钮权限
- 数据范围

推荐菜单模块仍属于同一个管理平台：

- 工作台
- 后台用户
- 角色管理
- 渠道管理
- 供应商管理
- 平台商品
- 订单管理
- 财务管理
- 风控管理
- 技术支持
- 投诉管理
- 审计日志

这些是管理平台里的菜单模块，不是多个登录平台。

## 3.3 路由守卫

管理平台至少需要两层守卫：

### 第一层：是否已登录

- 无 token 跳转登录页

### 第二层：是否有对应菜单 / 页面权限

- 校验角色
- 校验权限码
- 校验菜单码

不满足时：

- 跳转 `403`
- 提示无权限访问

---

## 4. 渠道管理平台前端实现要求

## 4.1 单独登录态

渠道管理平台应单独维护：

- 登录页
- 用户态
- 权限态
- 请求拦截器

不要和管理平台共用同一个用户 store。

## 4.2 数据边界

渠道平台前端不得实现：

- 手工切换 `channelId`
- 查看其他渠道数据

正确做法：

- 登录后默认只查看当前渠道自己的数据
- 权限边界以后端认证结果为准

## 5. 工程实现建议

在保持总体架构不变的前提下，建议前端只拆成两个应用：

```text
apps/
  admin-web
  portal-web

packages/
  shared-api-types
  shared-api-client
  shared-ui
  shared-utils
```

建议共享：

- 基础请求封装
- 类型定义
- 通用 UI 组件
- 工具方法

注意：

- 管理平台与渠道管理平台的登录态、权限态分别维护
- 共享的是基础能力，不是把两个平台揉成同一套用户体系

---

## 6. 与 `api.json` 的同步方式

建议流程：

1. 后端维护最新版 `api.json`
2. 前端根据 `api.json` 生成类型
3. 统一生成共享 API Client
4. 管理平台和渠道管理平台共同依赖共享 API 类型

## 8. 给开发工程师的提示词模板

## 8.1 示例：管理平台角色菜单控制

```text
你是一名前端工程师，请基于当前后台管理系统实现角色菜单控制。

项目背景：
- 当前系统只有一个内部管理平台
- 内部员工统一通过管理平台登录
- 不允许再拆出多个内部登录平台

目标：
- 登录后根据角色和权限展示不同菜单和功能

输入资料：
- api.json
- 现有 /admin/auth/login 与 /admin/auth/me 接口
- 现有路由和菜单结构

业务规则：
- 超级管理员、运营、财务、风控、技术支持都属于同一个管理平台
- 他们看到的只是菜单和功能不同，不是平台不同

技术要求：
- 增加角色与权限初始化逻辑
- 增加菜单过滤逻辑
- 增加页面级权限守卫

验收标准：
- 不同角色登录后看到不同菜单
- 但都属于同一个管理平台
```

## 8.2 示例：渠道管理平台登录与数据隔离

```text
你是一名前端工程师，请实现渠道管理平台登录与数据隔离。

项目背景：
- 当前系统只有一个渠道管理平台
- 外部渠道统一通过该平台登录
- 每个渠道只能查看自己的数据

目标：
- 完成登录页、用户态初始化、数据查询封装

输入资料：
- api.json
- 现有 /portal/auth/login 与 /portal/me 接口

业务规则：
- 前端不能通过切换 channelId 查看其他渠道数据
- 页面默认展示当前登录渠道的数据

技术要求：
- 独立 portal 用户态
- 独立 portal 请求封装
- 权限边界由后端认证结果决定

验收标准：
- 渠道登录后只能看到自己的数据
- 不能访问内部管理平台
```

## 9. 实施优先级

建议按以下顺序推进：

1. 先把架构固定为“一个管理平台 + 一个渠道管理平台”
2. 先完成管理平台角色菜单控制
3. 再完成渠道管理平台登录与数据边界
4. 最后接入平台商品配置层与投诉管理模块

---

## 10. 一句话结论

根据当前 `api.json`，前端正确做法不是拆更多平台，而是：

- 内部员工统一登录管理平台
- 外部渠道统一登录渠道管理平台
- 管理平台内部通过角色和权限控制菜单与功能
